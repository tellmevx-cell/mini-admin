# 租户资源配额闭环需求文档

## 背景

租户套餐已经可以配置最大用户数和最大存储空间，但这两个字段目前只用于展示，用户新增、批量导入和文件上传不会执行限制。文件记录也尚未记录租户归属，无法准确统计租户存储用量。

## 目标

- 让套餐用户数和存储空间配额真正约束租户写入行为。
- 平台管理员能在租户列表查看每个租户的实时资源用量。
- 补齐文件租户归属与访问边界，同时兼容现有平台文件。
- 并发写入时以租户为单位串行校验，避免普通并发绕过配额。

## 业务规则

- `MaxUsers = 0` 或 `MaxStorageMb = 0` 表示对应资源不限额。
- 用户数统计租户下全部账号，包括已停用账号；删除账号后释放名额。
- 存储用量统计租户下仍存在的文件记录字节数；删除文件后释放空间。
- 平台账号不受租户配额限制。
- 未绑定套餐的租户按不限额处理，避免破坏历史租户。
- 套餐降级后允许保留已有数据；已经超额时禁止继续新增对应资源。
- 创建租户时生成的首个租户管理员计入用户用量。

## 功能范围

- 单个新增用户前校验用户配额。
- 用户导入预检和确认导入时校验本次有效用户数量。
- 文件上传前预检并在文件记录落库前再次校验存储配额。
- 文件上传落库失败时删除已经写入存储介质的文件，避免孤儿文件。
- `ManagedFile` 增加可空 `TenantId`；旧数据保持 `null`，作为平台文件。
- 租户用户只能查询、下载、标记和删除本租户文件；平台用户保持全局管理能力。
- 平台租户列表显示用户数和存储空间的已用值、上限及超额状态。
- 租户套餐页面明确展示 `0 = 不限额`。

## 不做范围

- 不实现计费、订单、自动扩容或套餐购买。
- 不按文件状态区分计费，只要文件记录存在就计入存储用量。
- 不迁移无法可靠判断归属的历史平台文件。
- 不新增与平台资源配额无关的业务模块。

## 权限与安全

- 继续复用现有用户、文件、租户管理权限，不新增菜单。
- 配额校验必须从服务端执行，不能依赖前端提示。
- 配额错误返回明确业务消息，不暴露数据库和存储实现细节。
- 租户文件查询和单文件操作都必须校验 `TenantId`。

## 数据流转

```mermaid
flowchart LR
    A["租户写入请求"] --> B["配额服务"]
    B --> C["锁定当前租户"]
    C --> D["读取套餐与实时用量"]
    D --> E{"是否还有额度"}
    E -->|是| F["执行用户或文件写入"]
    E -->|否| G["返回配额不足"]
    F --> H["刷新租户列表用量"]
```

## 验收标准

- [x] 平台用户新增账号和上传文件不受租户配额影响。
- [x] 租户达到用户上限后，单个新增和批量导入均被拒绝。
- [x] 租户达到存储上限后，文件上传被拒绝且不产生文件记录或孤儿文件。
- [x] 删除租户用户或文件后，对应额度立即释放。
- [x] 两个租户的用户和文件用量互不影响。
- [x] 租户无法查询、下载、标记或删除其他租户文件。
- [x] 租户列表正确显示用户和存储用量，`0` 上限显示为不限额。
- [x] MySQL 迁移兼容旧 `mini_files` 数据，旧记录不被删除或错误归属。
- [x] 后端专项测试、全量测试及前端生产构建通过。
