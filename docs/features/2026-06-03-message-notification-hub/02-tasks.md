## TDD 计划

1. 先为工作流通知链路补失败测试：
   - 发起审批通知审批人
   - 审批通过通知发起人
   - 转办通知新处理人
   - 抄送节点通知被抄送人
2. 再为消息投递记录查询写失败测试。
3. 红灯确认后实现通用消息服务和投递记录查询。
4. 后端绿灯后升级前端页面结构和接口。
5. 最后跑工作流测试、通知相关测试、整体构建和接口烟测。

## 实现注意

- 工作流通知标题和内容要能让用户一眼看懂，不使用纯技术字段名。
- 同一用户同一来源的通知仍保持幂等去重，避免重复刷屏。
- 通知中心升级后，顶部铃铛的轻量能力保持兼容。
- 投递记录已覆盖邮件与 Webhook 通道，Webhook 默认关闭，配置地址后再实际推送。
- 自动重试默认最多重试 3 次，每批最多处理 50 条，可在定时任务中手动运行或等待后台调度。
- 个人订阅偏好只覆盖当前用户，且不能强制打开全局策略已经关闭的通道。
- 工作流抄送使用独立 `mini_workflow_cc_records` 记录阅知状态，历史 `Cc` 流转日志会在表初始化时回填为未读抄送记录。
- 本地 Node 环境实际可用版本为 `22.21.1`，因此将 `frontend/vue-vben-admin/.node-version` 调整为 `22.21.1`，与当前机器环境保持一致。
- 上线前按 `04-smoke-runbook.md` 执行一次手工冒烟即可，暂不继续增加催读、加签、代理审批、统计大屏等扩展能力。
