# SaaS 租户

MiniAdmin 提供 SaaS 租户基础能力，用于让同一套后台服务多个租户，并控制租户状态、套餐和初始化。

## 核心概念

| 概念 | 说明 |
| --- | --- |
| 租户 | 使用系统的客户或组织 |
| 租户编码 | 登录和识别租户的稳定编码 |
| 租户套餐 | 控制租户可用菜单、用户上限、存储容量等 |
| 租户管理员 | 租户初始化后默认创建的管理员账号 |
| 当前租户 | 请求上下文中的租户身份 |

## 登录识别

租户用户登录后，JWT 中会携带租户信息。后续请求通过当前租户上下文过滤数据。

如果租户被禁用：

- 新登录会被拒绝。
- 已有 token 的后续请求会被拦截。

## 租户套餐

租户套餐用于控制菜单和资源上限。

常见场景：

- 不同租户可见不同菜单。
- 不同套餐限制最大用户数。
- 不同套餐限制存储容量。

## 资源配额

套餐中的最大用户数和存储容量会在服务端真实生效，不只是展示字段。平台管理员可以在“平台管理 / 租户管理”的资源用量列查看每个租户的实时用户数、文件存储量、套餐上限和超额状态。

配额规则：

- 最大用户数或存储容量填写 `0` 表示不限额。
- 用户数包含租户下已启用和已停用的全部账号，租户管理员也会计入用量。
- 存储容量按租户仍存在的文件记录统计；删除用户或文件后会立即释放对应额度。
- 单个新增、批量导入和文件上传都会在后端校验，超限时接口返回 `409 Conflict` 和明确原因。
- 用户或文件的并发写入会按租户串行完成最终校验，不能通过并发请求绕过上限。
- 套餐降级不会删除已有数据；如果现有用量已经超额，只阻止继续新增对应资源。
- 平台账号和没有绑定套餐的历史租户按不限额处理。

文件上传会先做快速预检，再在文件记录入库前进行事务内最终校验。如果最终校验或落库失败，系统会尝试删除已经写入存储介质的对象，避免产生孤儿文件。

升级旧数据库时，`mini_files` 会增加可空的 `TenantId` 和索引。无法可靠判断租户归属的历史文件保持为空，并继续作为平台文件管理；新上传的租户文件会自动写入当前租户标识。

## 用量看板与配额预警

租户管理员进入“工作台”后，可以直接查看当前套餐、用户账号和文件存储的实时用量。每项资源都显示已用量、上限、使用比例和风险状态，并可快捷进入用户管理或文件管理释放资源。

系统统一使用以下状态：

| 状态 | 条件 | 行为 |
| --- | --- | --- |
| 不限额 | 套餐上限为 `0` | 只展示用量，不发送预警 |
| 正常 | 使用率低于 `80%` | 不发送预警 |
| 预警 | 使用率达到 `80%` 且低于 `100%` | 通知租户管理员 |
| 已耗尽 | 使用率达到或超过 `100%` | 发送严重通知，新增资源仍由配额服务拦截 |

预警任务名为“租户资源配额预警扫描”，默认每天执行一次，也可以在“系统监控 / 定时任务”中手动执行。任务只扫描启用且未过期的租户，并在执行日志中记录异常租户、资源类型、使用率、接收人数和通知数量。

通知会写入消息中心，并通过 SignalR 实时更新租户管理员的消息和未读数量。系统只在首次进入预警/耗尽状态或风险状态发生变化时通知；同一状态重复扫描不会重复发送。资源恢复正常后，再次达到阈值时可以重新通知。

默认 Scriban 模板代码为 `TenantQuota.Warning`，可在消息模板中心修改全局模板。租户也可以基于现有模板覆盖机制维护自己的文案和跳转地址。

当前租户用量查询由动态 API 提供：

```http
GET /tenant/resource-usage
```

该接口要求登录，并且只根据服务端当前租户上下文返回本租户用量，不接受客户端传入任意租户标识。平台上下文不返回租户用量，平台管理员应在“平台管理 / 租户管理”查看所有租户的状态和最近预警时间。

预警状态持久化在 `mini_tenant_resource_quota_warnings` 表中，每个租户和资源类型只有一条记录。表中保存最近扫描用量、状态、通知轮次和最近通知时间，通过唯一索引和消息来源标识保证重复扫描幂等。MySQL 使用增量迁移升级，InMemory 环境可直接运行测试。

## 租户生命周期

租户生命周期在已有到期登录校验的基础上，补齐主动提醒、自动状态流转、会话失效、续期和历史追踪。平台管理员在“平台管理 / 租户管理”可以直接看到长期有效、有效、30 天内、7 天内、1 天内和已到期标识。

### 到期提醒

内置“租户生命周期扫描”任务，默认每天执行一次，也可以在“系统监控 / 定时任务”中手工运行。任务只扫描启用且 30 天内到期的租户：

| 距离到期时间 | 接收人 | 行为 |
| --- | --- | --- |
| 30 天内 | 启用的租户管理员 | 发送一次站内提醒 |
| 7 天内 | 启用的租户管理员 | 再发送一次更紧急的提醒 |
| 1 天内 | 启用的租户管理员 | 发送严重级别提醒 |
| 已到期 | 平台管理员 | 状态改为 `Expired`，停止租户访问并发送平台通知 |

同一租户、同一到期时间、同一提醒档位只会通知一次。续期后到期时间变化，会自然形成新的提醒周期。如果扫描时没有可用的租户管理员，系统不会误记为已提醒，后续扫描仍会重试。

提醒复用消息中心和 SignalR 实时推送，默认模板代码为：

- `TenantLifecycle.ExpiryReminder`：租户临期提醒。
- `TenantLifecycle.Expired`：租户自动过期通知。

### 自动过期与会话安全

到期后系统会把数据库中的租户状态真实更新为 `Expired`，而不只是在列表中临时计算。随后会统一刷新该租户用户安全戳、清除授权缓存并将在线会话置为离线，因此已经签发的 token 也不能继续访问。

被手工停用的租户不会被生命周期任务自动启用。过期租户不能直接点击“启用”，必须先续期，避免出现状态看似启用但仍因到期时间被拒绝的矛盾状态。

### 续期与历史

平台管理员点击“续期”后选择新的未来到期时间，并决定是否同时恢复租户访问。续期会使旧会话失效，租户用户需要重新登录。套餐调整仍在现有“编辑租户”入口完成，续期接口不耦合订单或套餐购买。

点击“记录”可以查看以下生命周期事件：

- 租户创建和资料更新。
- 套餐调整和到期时间调整。
- 启用、停用、临期提醒和自动过期。
- 手工续期及操作人、来源、状态变化和时间。

相关接口：

```http
GET  /platform/tenant/{id}/lifecycle-records
POST /platform/tenant/{id}/renew
```

历史数据保存在 `mini_tenant_lifecycle_records`。`DeduplicationKey` 唯一索引用于保证提醒幂等；普通人工事件不使用该键，因此完整保留每次操作记录。

## 数据隔离

租户数据隔离的原则：

- 业务表如果属于租户，应包含 `TenantId`。
- 查询时默认按当前租户过滤。
- 平台管理员可以在明确场景下跨租户管理。
- 不要让普通租户用户传入任意 `TenantId` 查询数据。

## 二开建议

新增业务模块时先判断：

1. 这个模块是否属于租户数据。
2. 是否需要 `TenantId`。
3. 是否需要租户内唯一约束。
4. 是否需要套餐控制入口。
5. 是否需要平台管理员跨租户维护。

如果不确定，默认把业务数据设计为租户数据更安全。
