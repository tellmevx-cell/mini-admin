# 消息中心

消息中心负责把系统事件和用户通知统一管理起来。它不是单纯的铃铛列表，而是包含模板、策略、订阅、站内信和投递记录的一套通知平台。

## 能力组成

| 能力 | 说明 |
| --- | --- |
| 站内信 | 用户登录后台后看到的消息 |
| 顶部铃铛 | 快速查看未读消息 |
| 通知模板 | 维护标题、内容和跳转链接 |
| 通知策略 | 控制事件是否通知、通知到哪些通道 |
| 订阅偏好 | 用户个人接收偏好 |
| 投递记录 | 追踪邮件、Webhook 等外部通道发送结果 |
| 失败告警 | 外部投递失败时通知管理员 |

## 事件模型

每条通知通常包含：

- 事件类型。
- 来源类型。
- 来源 ID。
- 接收人。
- 标题。
- 内容。
- 跳转链接。
- 已读状态。

工作流事件已经接入消息中心，包括待办、通过、驳回、撤回、转办、催办和抄送。

## 模板渲染

模板用于避免通知文案散落在业务代码中。

建议：

- 标题短，能看出事件。
- 内容包含关键上下文。
- 链接保留跳转参数。
- 修改模板后触发一次实际事件验证。

## 策略和订阅

全局通知策略决定通道是否可用，个人订阅偏好只能在全局允许范围内调整。

规则：

- 全局关闭的通道，个人不能强行打开。
- 个人恢复默认后，重新跟随全局策略。
- 站内信是后台最基础的通知通道，建议默认保留。

## 投递记录

邮件和 Webhook 可能因为配置、网络或对方服务失败。投递记录用于排查这些问题。

常见状态：

| 状态 | 含义 |
| --- | --- |
| Sent | 已发送 |
| Failed | 发送失败 |
| Skipped | 因配置或策略跳过 |
| Pending | 等待发送或等待重试 |

## 二开接入建议

新增业务通知时：

1. 定义事件类型。
2. 增加默认通知策略。
3. 增加默认模板。
4. 业务动作发生时调用通知服务。
5. 在消息跳转链接中携带业务定位参数。
6. 增加最小测试，验证通知生成和模板渲染。
